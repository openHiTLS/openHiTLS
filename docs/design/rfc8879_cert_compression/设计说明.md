# RFC8879 证书压缩

[RFC8879](https://datatracker.ietf.org/doc/html/rfc8879) 定义了 TLS 证书压缩扩展 `compress_certificate`，用于降低证书链传输开销。
当前实现以 [RFC8446](https://datatracker.ietf.org/doc/html/rfc8446) TLS 1.3 握手流程为主协议基线，同时显式覆盖
[RFC5246](https://datatracker.ietf.org/doc/html/rfc5246) TLS 1.2 及更低版本的兼容边界，即协商到 TLS 1.2 或更早版本时必须忽略该扩展。

## 设计思路

当前 RFC8879 实现由四部分组成：

1. 配置层

对外提供证书压缩能力开关、算法列表、压缩阈值、最大解压长度等配置接口，配置入口定义在
`include/tls/hitls_config.h` 和 `include/tls/hitls.h`，具体实现位于
`tls/config/src/config_feature.c` 和 `tls/cm/src/conn_ctrl.c`。

2. 扩展协商

客户端在 TLS 1.3 ClientHello 中按配置打包 `compress_certificate` 扩展，服务端在解析 ClientHello 后根据本地配置和
对端算法列表选择协商算法。相关实现位于：

- `tls/handshake/pack/src/pack_extensions.c`
- `tls/handshake/parse/src/parse_extensions_server.c`
- `tls/handshake/common/src/hs_common.c`
- `tls/handshake/recv/src/recv_client_hello.c`

3. 证书发送

TLS 1.3 下，如果双方成功协商证书压缩，且证书消息长度达到阈值，则服务端或客户端发送
`CompressedCertificate`；否则回退为普通 `Certificate`。相关实现位于：

- `tls/handshake/pack/src/pack_certificate.c`
- `tls/handshake/send/src/send_certificate.c`

4. 证书解析

对端收到 `CompressedCertificate` 后，先校验算法、长度和最大解压长度，再解压并复用现有证书解析逻辑。
同时，TLS 1.2 及以下版本不允许接收该握手消息类型。相关实现位于：

- `tls/handshake/parse/src/parse_certificate.c`
- `tls/handshake/parse/src/parse.c`

## 当前实现现状

基于当前分支最新代码，RFC8879 相关实现还补充了以下收敛修正：

1. `identity` 压缩算法的 encode/decode 路径已去除对 `securec` 的依赖，避免在 `no-provider` 构建路径下引入额外编译失败。
2. 为满足社区 codecheck 要求，RFC8879 新增的数据复制路径未继续使用 `memcpy`，而是保留长度校验后进行逐字节复制。
3. `CompressedCertificate` 解析阶段已经覆盖：
   - `compressed_length` 异常
   - `uncompressed_length == 0`
   - `uncompressed_length > maxUncompressedLen`
   - 压缩体损坏但长度字段合法

这些修正没有引入额外兼容层，仍然保持在现有 TLS 握手主路径内闭环处理。

## Demo 定位

当前 demo 的作用是：

1. 作为对外接口和集成方式的使用样例。
2. 作为构建后可执行文件的基础联通性检查。
3. 在 `testcode/script/execute_sdv.sh` 中作为附加 smoke check 被统一执行。

从执行脚本实现可以看到，`run_demos()` 会直接遍历 `testcode/demo/build` 下的可执行文件并运行 `client/server`
与 `tlcp_client/tlcp_server` 组合，因此 demo 当前承担的是“示例 + 运行冒烟”的角色。

结合近期脚本修复，当前 demo 执行还有两点约束：

1. `run_demos()` 已增加服务端超时等待与异常回收，避免 demo 服务端进程异常时长期阻塞整条 SDV 流水线。
2. demo 执行入口已过滤 `CMakeCache.txt`、`Makefile` 等构建辅助文件，避免非 demo 产物被误当成可执行文件运行。

## 当前测试覆盖

当前 RFC8879 相关的主要断言测试位于：

- `testcode/sdv/testcase/tls/consistency/tls13/test_suite_sdv_frame_tls13_consistency_rfc8446_extensions_1.c`

当前已覆盖的关键场景如下：

1. TLS 1.3 成功协商证书压缩，并发送 `CompressedCertificate`。
2. TLS 1.3 双方算法不匹配，回退发送普通 `Certificate`。
3. TLS 1.3 压缩证书中的 `compressed_length` 被篡改，客户端解析失败并发送告警。
4. TLS 1.2 协商完成后，服务端忽略 `compress_certificate` 扩展，不发送压缩证书。
5. TLS 1.2 协商完成后，客户端若收到伪造的 `CompressedCertificate`，返回
   `HITLS_MSG_HANDLE_UNEXPECTED_MESSAGE` 并发送 `unexpected_message` 告警。
6. TLS 1.3 ClientHello 中重复出现 `compress_certificate` 扩展。
7. TLS 1.3 `compress_certificate` 扩展中的算法列表长度非法。

从实现路径看，上述测试已经触达：

1. 配置接口与默认值。
2. ClientHello 扩展打包和服务端解析。
3. 算法协商与回退。
4. `CompressedCertificate` 发送逻辑。
5. `CompressedCertificate` 解析逻辑。
6. TLS 1.2 对该扩展和握手消息类型的边界处理。
